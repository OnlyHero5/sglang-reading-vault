# 性能指标与实验方法

## 你为什么要读

一个性能数字只有在“测量边界、workload、版本、硬件、正确性”都明确时才有意义。把 server 内部 TTFT 与客户端 TTFT 混比、把 output throughput 与 total-token throughput 混比，或用单个 kernel 变快推断端到端变快，都会得到看似精确的错误结论。

## 第一原则：先定义问题和正确性

实验前先写：

```text
问题：我要解释哪个现象？
假设：改变 X 会通过机制 M 影响指标 Y。
正确性门槛：输出、梯度、版本或请求完成语义必须满足什么？
独立变量：本轮只改什么？
固定项：模型、revision、硬件、软件、输入分布、并发、随机性。
```

性能优化若改变了 mask、causal 对齐、采样、丢请求或训练算法，就不能只报“更快”。

## Serving 指标：先写观察点

| 指标 | 定义 | 常见陷阱 |
|------|------|----------|
| E2E latency | 客户端发出到最终完成 | 含网络、排队、生成长度；不同长度不能直接平均比较 |
| TTFT | 请求到首个可见 token | client/server 观察点不同；受 queue、prefill、stream flush 影响 |
| inter-token latency | 相邻可见 token 的间隔分布 | 不等于单一平均 TPOT；speculative/批量 flush 会改变形态 |
| TPOT | 常按首 token 后的生成时间除以后续 token 数 | 需声明公式、是否含最后 flush 和零/一 token 请求 |
| request throughput | 单位时间完成请求数 | 输出长度变化会让数字失真 |
| token throughput | prompt/output/total token 每秒 | 三种口径必须写清，不能只写 tok/s |
| goodput | 满足给定 SLA 的有效完成率/吞吐 | 必须给 TTFT/TPOT/E2E 阈值和失败规则 |
| KV/cache 指标 | usage、hit、evict、retract | allocator/layout 不同，百分比不一定横向可比 |

至少报告 P50/P90/P99、失败/超时/abort 数和长度分桶。只报告均值会隐藏排队与尾延迟。

## 训练与 RL 闭环指标

把一轮时间拆开：

```text
data wait / rollout / RM-filter / train wait / forward-backward-optimizer
/ weight sync / checkpoint / eval / offload-onload
```

| 类别 | 例子 | 解释边界 |
|------|------|----------|
| 吞吐 | samples/s、有效 response tokens/s、train tokens/s | 是否含过滤/失败/重试？ |
| 算法状态 | reward、loss、KL、entropy、clipfrac、grad norm | 指标口径与 loss reducer 是否一致？ |
| 陈旧性 | sample age、生成权重版本与训练版本差 | 框架是否真的记录 version；如何重建？ |
| 资源 | GPU active、memory、通信时间、MFU | MFU 依赖 FLOP 模型，不是通用效率真值 |
| 稳定性 | NaN、OOM、actor/engine restart、丢组 | 吞吐不能排除静默数据损失 |

同步与异步方案比较时，除了吞吐必须报告 sample staleness 与算法结果；否则可能只是用更旧的数据换更高重叠。

## Kernel 指标：局部与端到端分开

| 层级 | 观测 | 注意 |
|------|------|------|
| kernel | CUDA event duration、HBM bytes、achieved bandwidth、Tensor Core 指令、register/shared memory | GPU 异步；host timer 要 synchronize |
| operator | 输入准备 + dispatch + kernel + combine | split/transpose/workspace 可能在 kernel 外 |
| model step | 多算子、通信、graph replay | 单 kernel 加速可能被其他阶段吞没 |
| end-to-end | request 或 train iteration | 还受排队、CPU、网络、数据与同步影响 |

Occupancy 高不等于快：它是资源驻留能力，不直接表示有效工作、带宽或 Tensor Core 利用。duration 下降也要同时检查数值误差、deterministic 和 workspace。

## 合格实验记录模板

```text
问题 / 假设：
代码基线：commit、dirty diff、依赖版本
硬件：GPU 型号/数量、互联、CPU、内存、驱动/CUDA
模型与数据：revision、dtype、shape/长度分布、采样参数
拓扑与配置：DP/TP/PP/CP/EP、batch、并发、cache、graph
独立变量：
固定项与已知混杂：
正确性验收：reference、容差、完成/失败计数
预热：次数和停止条件
采样：重复次数、时间窗口、随机种子
指标定义与观察点：
结果：分布、方差/置信区间、原始日志位置
结论与不可外推范围：
```

### 常见混杂因素

- 首轮编译、autotune、CUDA graph capture 和 page cache；
- 动态输入长度、输出长度、prefix hit 和 request arrival；
- GPU 时钟、温度、共享节点噪声和后台任务；
- allocator 缓存、模型 reload、checkpoint 或 eval 插入；
- 错误重试、过滤、abort 后“少做工作”造成的假加速。

## Trace、Profile、Metric、Log 的分工

| 工具 | 最擅长回答 | 不能单独回答 |
|------|------------|--------------|
| trace | 一条请求/样本跨阶段的 critical path | kernel 微观瓶颈、长期分布 |
| profiler | CPU/GPU 区段内部算子、stream、memory | 业务身份和长期健康 |
| metric | 长时间窗口的速率、分位数、容量趋势 | 某次状态为何跳变 |
| log/event | 离散决策、异常和版本提交 | 无聚合时的总体分布 |

正确流程常是：metric 发现异常 → trace 定位阶段 → profiler 深挖执行 → log/源码解释决策。

## 复盘

- 指标名相同，不代表观察点和公式相同。
- 正确性、完成数量和版本一致性必须先于性能结论。
- kernel、operator、model step、end-to-end 是四个不同层级。
- 好实验不仅报告结果，还报告为什么可以相信、为什么不能外推。
