# 性能指标与实验方法

## 学习目标

读完后，你应该能设计一个有基线、只改变一个变量、记录 workload 和硬件条件的实验，并能区分延迟、吞吐、利用率和正确性。

## Serving 指标

| 指标 | 含义 | 常见影响因素 |
|------|------|--------------|
| TTFT | 请求到首 token | 排队、prefill、prefix hit、网络 |
| TPOT | 相邻输出 token 时间 | decode batch、KV 带宽、overlap |
| throughput | 单位时间 token/request 数 | batch、并行度、请求分布 |
| goodput | 满足 SLA 的有效吞吐 | P99、失败率、超时 |
| KV usage | KV pool 使用比例 | 并发、长度、page、retract |

## 训练指标

- samples/tokens per second
- rollout time、reward time、train time、weight sync time
- loss、KL、entropy、clip fraction、gradient norm
- GPU memory、MFU、通信占比
- sample age 和 weight version lag

## Kernel 指标

- kernel duration
- achieved bandwidth
- Tensor Core / SM utilization
- occupancy、register、shared memory
- HBM read/write bytes
- 数值误差与 deterministic 行为

## 一个合格实验的记录模板

```text
问题：关闭 overlap 是否降低吞吐但更易调试？
基线：默认配置
变量：仅改变 --disable-overlap-schedule
固定项：模型、GPU、请求长度分布、并发、随机种子
观测：TTFT P50/P99、TPOT、throughput、KV usage、错误率
重复：预热后至少多轮运行
结论：报告分布和置信范围，不只报告最好一次
```

## Trace、Profile、Metric 的分工

| 工具 | 回答的问题 |
|------|------------|
| trace | 一条请求或样本把时间花在哪里 |
| profiler | 某段 CPU/GPU 执行内部谁最慢 |
| metric | 系统长期是否偏离健康区间 |
| log | 某次状态迁移为什么发生 |

## 运行验证

选择 [[SGLang服务实验]] 或 [[FlashAttention性能实验]]，先写实验记录再运行命令。

预期：结论必须包含 workload、硬件、版本和配置；不能把单次数字推广成框架通用阈值。

## 复盘

- 性能数字离开 workload 没有可比性。
- 正确性验证先于性能比较。
- Trace、profile、metric 和 log 解决不同层次的问题。
