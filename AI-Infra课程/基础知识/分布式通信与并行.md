# 分布式通信与并行

## 学习目标

读完后，你应该能解释一个 rank 属于哪些通信组，分清 DP、TP、PP、CP、EP，并能用 collective 顺序定位常见 hang。

## 五种并行维度

| 维度 | 切分对象 | 主要通信 |
|------|----------|----------|
| DP | 不同样本副本 | 梯度 all-reduce / reduce-scatter |
| TP | 单层 tensor | all-reduce、all-gather |
| PP | 不同层 | stage 间 activation send/recv |
| CP | 长序列上下文 | sequence 分片与 gather |
| EP | MoE experts | token dispatch / all-to-all |

同一个进程可能同时拥有多个 rank 身份。不要只记录 global rank；还要记录它在每个 group 中的 local rank。

## Collective 是群体协议

Broadcast、all-reduce、all-gather、reduce-scatter、all-to-all 都要求组内 rank 以一致顺序参与。某个 rank 跳过一次调用，其他 rank 通常表现为 hang，而不是清晰异常。

排查顺序：

1. group 成员和 world size 是否一致。
2. 每个 rank 是否进入相同 collective。
3. tensor shape、dtype、device 是否一致。
4. collective 顺序是否一致。
5. 是否有更早的 rank 异常导致其他 rank 等待。

## Slime 与 SGLang 中的映射

- Slime PlacementGroup 决定哪些 GPU 给 rollout、actor 和 critic。
- Megatron 初始化 TP/PP/DP/CP/EP 通信组。
- SGLang Scheduler 只让入口 rank 拉请求，再广播给相关 rank。
- 权重同步同时使用 Ray 控制面和 NCCL tensor 数据面。

## 一致性屏障

权重同步不只是传 tensor。正确顺序通常是：

```text
暂停生成 -> 清理旧 KV -> 所有训练 rank 对齐 -> 发送 metadata
-> collective 传 tensor -> engine 加载 -> 更新版本 -> 恢复生成
```

任何一步提前恢复，都可能让请求看到混合版本。

## 运行验证

阅读 [[Slime-分布式权重同步-数据流]]，画出 Ray metadata 通道和 NCCL payload 通道。

预期：两条通道必须对同一个 bucket、shape 和 weight version 达成一致；只看到 NCCL broadcast 不代表 engine 已完成可见更新。

## 复盘

- 并行维度描述的是切分对象，不是 GPU 数量的同义词。
- collective hang 优先检查参与者和顺序。
- 控制面与数据面经常使用不同通信机制。
