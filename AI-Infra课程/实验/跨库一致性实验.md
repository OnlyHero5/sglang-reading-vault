# 跨库一致性实验

## 学习目标

建立一张能把 Slime Sample、SGLang request、attention backend 和权重版本关联起来的观测表，验证训练闭环不是三个独立日志文件的时间戳拼接。

本实验还要验证一个容易被忽略的事实：有些关联是源码自动保存的，有些只是“理论上应该能查到”。缺失字段必须显式记录为观测缺口，不能编造一一对应关系。

## 关联字段

至少记录：

| 字段 | 来源 |
|------|------|
| rollout step | `generate_rollout(..., rollout_id)` 参数与 RolloutManager 日志 |
| sample id / prompt group | `Sample.index` / `Sample.group_index` |
| loss 聚合 id | `Sample.rollout_id` |
| request id | Sample trace 中的 `sglang_request_id`；标准 Sample 没有 `rid` 字段 |
| engine / rank | SGLang topology |
| weight versions | `Sample.weight_versions` 与 updater/engine version |
| input/output token count | tokenizer / Sample |
| TTFT / TPOT | SGLang metrics |
| reward / KL / loss | Slime training |

## 阶段 A1：标准库静态契约实验

这一步不导入 Slime，可在缺少 Torch/HTTPX 的维护机上运行：

```powershell
@'
import ast
from pathlib import Path

tree = ast.parse(Path("slime/slime/utils/types.py").read_text(encoding="utf-8"))
sample_cls = next(n for n in tree.body if isinstance(n, ast.ClassDef) and n.name == "Sample")
fields = [n.target.id for n in sample_cls.body if isinstance(n, ast.AnnAssign)]
trace_text = Path("slime/slime/utils/trace_utils.py").read_text(encoding="utf-8")

print("has_weight_versions=", "weight_versions" in fields)
print("has_weight_version=", "weight_version" in fields)
print("has_rid=", "rid" in fields)
print("trace_maps_request_id=", 'attrs["sglang_request_id"] = meta["id"]' in trace_text)
'@ | python -
```

预期：复数字段和 trace 映射为 `True`，单数版本字段与 `rid` 为 `False`。

## 阶段 A2：完整 Slime 环境中的无 GPU行为实验

前提：当前 Python 环境能导入 Slime、Torch、HTTPX，且 Torch/NumPy ABI 匹配。从知识库根目录执行：

```powershell
$env:PYTHONPATH = (Resolve-Path "slime").Path
@'
from types import SimpleNamespace
from slime.utils.trace_utils import build_sglang_meta_trace_attrs, trace_span
from slime.utils.types import Sample

sample = Sample(index=12, group_index=3)
meta = {
    "id": "req-demo-001",
    "finish_reason": {"type": "stop"},
    "weight_version": "17",
    "prompt_tokens": 128,
    "completion_tokens": 8,
    "e2e_latency": 0.25,
}

with trace_span(sample, "sglang_generate") as span:
    span.update(build_sglang_meta_trace_attrs(meta))

sample.append_response_tokens(
    SimpleNamespace(sglang_speculative_algorithm=False),
    tokens=list(range(8)),
    log_probs=[-0.1] * 8,
    text="demo",
    meta_info=meta,
)

end_event = next(e for e in reversed(sample.trace["events"]) if e["type"] == "span_end")
print("sample_has_rid=", hasattr(sample, "rid"))
print("trace_request_id=", end_event["attrs"]["sglang_request_id"])
print("weight_versions=", sample.weight_versions)
print("status=", sample.status.value)
'@ | python -
```

预期：

```text
sample_has_rid= False
trace_request_id= req-demo-001
weight_versions= ['17']
status= completed
```

这一步证明 request id 默认属于 trace，版本属于 `weight_versions`，而不是虚构的 `Sample.rid` / `Sample.weight_version`。如果依赖导入失败，保留错误信息并以阶段 A1 作为静态替代。

## 基线运行

在可运行的 Slime 环境中运行一个最小闭环，并保留 Sample trace、rollout dump、SGLang server info 与训练指标。至少选一条 Sample，导出以下一行：

| rollout_step | sample.index | sample.group_index | sample.rollout_id | trace_id | sglang_request_id | weight_versions | prompt_tokens | completion_tokens | reward | KL |
|---:|---:|---:|---:|---|---|---|---:|---:|---:|---:|
| 42 | 12 | 3 | `null` 或聚合 id | … | … | `["17"]` | 128 | 64 | … | … |

无法运行完整闭环时，阶段 A1/A2 就是替代；不要填写无法由 dump/trace 证明的 engine、rank 或 kernel 名。

预期：保存了 trace 的 Sample 能反查 SGLang request id；终态 Sample 能看到 `weight_versions`。若 trace 未保存，应把 request id 标为“未观测”，而不是从时间戳猜测。

## 版本故障注入

仅在隔离环境延迟某个 rollout engine 的更新，或使用已有 fault-tolerance 测试模拟恢复。

预期：系统可能等待/恢复，也可能允许旧版本 Sample 继续进入 off-policy 训练；关键是 `weight_versions`、updater/engine 日志和训练配置能暴露事实，不能把旧版本样本无证据地称为 current-policy 数据。多轮 Sample 若出现 `['17', '18']`，必须解释一次样本为何跨版本，以及算法是否允许。

## KV 生命周期检查

针对实际 updater 路径，记录更新锁/暂停、cache flush 或失效动作、prefix hit 和首个请求版本。不同 distributed/tensor/disk 路径的动作顺序不完全相同，先从源码确认再设计观测。

预期：新权重生效后不继续复用依赖旧权重的 KV；cache hit 短暂下降是可解释现象。

## Kernel 证据

选择一条 rollout 请求，用 profiler 确认 attention backend 的实际 kernel。不要假设 SGLang 一定使用 FlashAttention；记录真实 backend 和硬件路径。

预期：跨库地图描述的是概念边界，实验记录必须说明实际选用 FlashInfer、Triton、FlashAttention 或其他实现。

## 通过标准

- [ ] 一张表分别记录 rollout step、`Sample.rollout_id`、request id、版本和训练指标。
- [ ] 能证明 request id 默认在 trace 中，而不是声称 Sample 自带 `rid`。
- [ ] 使用真实字段 `weight_versions`，并能解释列表为何可能含多个版本。
- [ ] 能解释当前 updater 的 quiescence/lock、cache 失效和 version commit；不套用统一暂停/flush 顺序。
- [ ] 能识别实际 attention backend，而不是按文档猜测。
- [ ] 能给出一次故障注入的症状、定位证据和恢复结果。
