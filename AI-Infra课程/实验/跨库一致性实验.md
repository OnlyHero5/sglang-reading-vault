# 跨库一致性实验

## 学习目标

建立一张能把 Slime Sample、SGLang request 和权重版本关联起来的观测表，验证训练闭环不是三个独立日志文件的拼接。

## 关联字段

至少记录：

| 字段 | 来源 |
|------|------|
| rollout_id | Slime RolloutManager |
| sample id / prompt id | DataSource / Sample |
| rid | SGLang request |
| engine / rank | SGLang topology |
| weight_version | rollout response 与 updater |
| input/output token count | tokenizer / Sample |
| TTFT / TPOT | SGLang metrics |
| reward / KL / loss | Slime training |

## 基线运行

运行一个最小闭环，导出上述字段。无法运行完整闭环时，可使用已保存 rollout dump 和 SGLang trace 做离线关联。

预期：每条 Sample 能反查生成它的请求和 weight version；缺少关联字段必须被记录为观测缺口。

## 版本故障注入

仅在隔离环境延迟某个 rollout engine 的更新，或使用已有 fault-tolerance 测试模拟恢复。

预期：系统应暂停、恢复或明确标记旧版本样本；不能静默把旧版本 Sample 混入当前 on-policy batch。

## KV 生命周期检查

权重更新前后记录 cache flush、prefix hit 和首个请求版本。

预期：新权重生效后不继续复用依赖旧权重的 KV；cache hit 短暂下降是可解释现象。

## Kernel 证据

选择一条 rollout 请求，用 profiler 确认 attention backend 的实际 kernel。不要假设 SGLang 一定使用 FlashAttention；记录真实 backend 和硬件路径。

预期：跨库地图描述的是概念边界，实验记录必须说明实际选用 FlashInfer、Triton、FlashAttention 或其他实现。

## 通过标准

- [ ] 一张表关联 rollout、request、版本和训练指标。
- [ ] 能解释权重更新期间为什么要暂停和 flush。
- [ ] 能识别实际 attention backend，而不是按文档猜测。
- [ ] 能给出一次故障注入的症状、定位证据和恢复结果。
