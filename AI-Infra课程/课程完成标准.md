# AI Infra 课程完成标准

## 你为什么要做这组验收

完成课程不等于浏览完链接。真正的通过标准是：能独立建立对象/版本模型，能从症状找到源码第一入口，能选择与结论强度匹配的验证，并明确尚未验证的环境边界。

## 证据等级

| 等级 | 你做了什么 | 可以声称什么 |
|------|------------|--------------|
| 静态定位 | 找到对象、调用点、分支和测试 | “当前源码存在此路径/契约” |
| CPU/纯 Python | 跑 reference、shape、contract、数学不变量 | “不依赖 GPU 的目标断言通过” |
| 单机 GPU | 跑 kernel/model/service 的数值与性能 | “当前 GPU/shape/workload 下通过” |
| 多进程/多机 e2e | 跑真实 topology、collective、恢复 | “当前集群配置的闭环通过” |

高等级不能由低等级自动推断。环境缺失时，用静态替代继续学习，但结论必须停在相应等级。

## 第一关：三张主线图

不看笔记画出：

1. **Serving：** HTTP → request state → schedule/forward view → token id → text delta → protocol chunk。
2. **RL：** prompt group → Sample → train ObjectRef → advantage/loss → optimizer → engine version。
3. **Attention：** Q/K/V → Python/C++ params → dispatch → tile/online softmax → O/LSE → backward state。

通过条件：每条箭头写 payload 或事件，每个框写所有者；不能只画框架名。

## 第二关：四本跨框架身份账

用一张表区分：

| 身份 | 作用域 | 创建者 | 何时结束 | 与其他身份如何关联 |
|------|--------|--------|----------|--------------------|
| API/internal request id |  |  |  |  |
| Sample `index/group_index` 与 `rollout_id` |  |  |  |  |
| KV slot/page/location |  |  |  |  |
| optimizer/updater/engine version |  |  |  |  |

通过条件：明确这些 id 相关但不相等；尤其不能把 `rollout_id` 当 weight version，不能把 batch index 当持久 KV 地址。

## 第三关：源码定位

任选一个外部 API 或 CLI 参数，完成：

- [ ] 找到输入定义、validation 和第一个运行时消费者。
- [ ] 沿一个核心对象找到创建、修改、跨边界发送和释放/完成位置。
- [ ] 找到至少一个配置分叉和一个异常/early-return 分支。
- [ ] 找到对应测试，并说明测试覆盖和盲区。
- [ ] 若引用代码，逐行核对且一卡只证明一个判断。

通过条件：结论不能只来自注释或函数名，必须结合调用点/测试/运行信号。

## 第四关：性能实验

选择 [[SGLang服务实验]] 或 [[FlashAttention性能实验]]，提交完整记录：

- 问题、机制假设和 correctness gate；
- commit/dirty diff、依赖、GPU/CPU/互联；
- 模型/shape/长度分布/dtype/采样参数；
- 独立变量与固定项；
- 预热、重复次数、原始数据；
- 指标公式和 client/server/kernel 观察点；
- 分布、失败数、资源数据和不可外推范围。

通过条件：不能只报最好一次、单个 utilization 或无 workload 的 tok/s。

## 第五关：RL 闭环验证

按环境选择：

- **静态/CPU：** 追踪同步与异步主循环，验证 Sample/contract、ratio/advantage 数学和 updater version 入口。
- **完整环境：** 运行 [[Slime闭环实验]]，记录 initial push、generate/train/update、样本数、group、loss 和 engine version。

通过条件：说清 rollout/old/current/reference policy；异步方案必须报告 sample staleness，而不只报告吞吐。

## 第六关：故障推演或注入

至少选择一类真实故障：

- 服务有 token id 但无文本；
- KV 不足导致等待/retract/abort；
- collective 某 rank 缺席；
- fan-out/group reward 错位；
- 权重更新中途失败或 engine 恢复后版本陈旧；
- FlashAttention causal/dropout/head-dim 条件错误。

交付格式：

```text
症状：
最可能失效的不变量：
源码第一入口：
操作/注入：
预期信号：
实际结果：
恢复与残留状态：
证据等级和环境限制：
```

没有可运行环境时做静态故障推演，明确哪些动态结论尚未验证。

## 第七关：十分钟复述

不看文档复述 [[从Prompt到新权重]]，必须包含：

- request、Sample/group、KV、policy version 四本身份账；
- prefill/decode 与 attention tile；
- reward→advantage→loss；
- 同步/异步的版本差别；
- 一处背压、一处 collective、一处失败恢复边界；
- 每个关键判断使用什么证据验证。

如果中途只能说“框架会处理”，回到相应主线补上对象、所有者、完成信号和失败状态。

## 最终交付清单

- [ ] 三张主线图。
- [ ] 四本身份账。
- [ ] 一份源码定位记录。
- [ ] 一份可复现实验记录。
- [ ] 一份 RL 闭环/静态替代记录。
- [ ] 一份故障推演或注入报告。
- [ ] 十分钟口述无“框架自动处理”空白段。

全部交付物都应注明版本与证据等级。完成数量不是目标；能解释、能证伪、能复现、能限定结论才是完成。

## 运行验证与最终判定

把交付物交给另一位读者复现：对方应能仅凭记录重跑静态命令/实验、定位原始日志，并区分“已通过”与“环境未覆盖”。若命令不可执行、预期不明确、结果无法追溯或结论超出证据等级，本课程验收不通过。
