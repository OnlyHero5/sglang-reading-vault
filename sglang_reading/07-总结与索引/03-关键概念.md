## 请求生命周期（concept:request-lifecycle）

**定义：** 一条生成请求从客户端到响应的完整七 hop 路径：HTTP/gRPC 入口 → TokenizerManager tokenize → Scheduler 组 batch → ModelRunner forward → Detokenizer 反 tokenize → 流式回传。

**Explain：** baseline HTTP 路径中，FastAPI `/generate` 把 JSON 反序列化为 `GenerateReqInput`，交给 TokenizerManager 的 async generator；主进程经 ZMQ 把 tokenized 请求送到 Scheduler 子进程，Scheduler 在 `event_loop_overlap` 里交替「组 batch + GPU forward」与「处理上一轮结果」。每个 decode step 的 token id 经 Detokenizer 子进程变回字符串，再 yield 给 HTTP 层写 SSE。gRPC 路径 hop 0–1 换成 Proto + Tonic + `RuntimeHandle`，hop 2 之后与 HTTP 共用 TokenizerManager 以下链路。

**Code：**

```python
        """A scheduler loop that overlaps the CPU processing and GPU computation."""

            Tuple[ScheduleBatch, Union[GenerationBatchResult, EmbeddingBatchResult]]
        ] = deque()
```

- 七 hop 逐步展开见 [[全链路请求追踪]]；gRPC 对称版见 [[全链路请求追踪-gRPC]]。
- PD 分离时 Decode 侧 hop 4 使用 `ForwardMode.PREBUILT` 跳过 prefill forward；投机解码时 hop 4 出现 `TARGET_VERIFY` / `DRAFT_EXTEND_V2` 交替。

## OpenAI 兼容层（concept:openai-compat）

**定义：** 将 OpenAI Chat Completions / Embeddings 等外部 API schema 映射为 SGLang 内部 `GenerateReqInput` / `EmbeddingReqInput`，再包装回 OpenAI 响应格式。

## SGLang gRPC Proto（concept:sglang-grpc-proto）

**定义：** `proto/sglang/runtime/v1/sglang.proto` 定义的三层 RPC：SGLang 原生 typed RPC、OpenAI JSON pass-through、Admin/Ops；gateway 与 server 共用同一 schema。

## gRPC Python 桥接（concept:grpc-py-bridge）

**定义：** Rust Tonic gRPC 服务通过 PyO3 调用 Python `RuntimeHandle`；响应 chunk 经 `chunk_callback` 回传 Rust mpsc channel，含背压与 abort 传播。

**Explain：** `RuntimeHandle` 在初始化时绑定 `TokenizerManager` 并 `auto_create_handle_loop`，保证所有 async 协程跑在 TM 专用 event loop 上。Rust 线程通过 `submit_*` 同步入口 `_submit_on_tm_loop` 投递协程；流式 generate 每个 chunk 经 `_send_with_backpressure` 等待 Rust 侧 `Pending` 状态解除，超时则 abort 请求，避免 Python 无限堆积。

**Code：**

```python
        ready_event = None
        try:

            if stream:
                async for chunk in gen:

                        chunk_callback,
                        ready_event,
                        chunk,
                        finished=finished,
                        timeout_abort_rid=obj.rid,
                    )
                    if finished or not keep_going:
                        return
```

```python
## 来源：python/sglang/srt/entrypoints/grpc_bridge.py L115-L153
    async def _send_with_backpressure(
        self,
        chunk_callback,
        ready_event: Optional[asyncio.Event],
        payload,
        *,
        timeout_abort_rid=None,
        **kwargs,
    ) -> bool:
        status = self._safe_callback(chunk_callback, payload, **kwargs)
        if status is None or self._is_closed_status(status):
            return False
        if not self._is_pending_status(status):
            return True

        if kwargs.get("finished"):
            return True
        if ready_event is None:
            return True

        try:
            await asyncio.wait_for(
                ready_event.wait(), timeout=self._BACKPRESSURE_TIMEOUT_S
            )
        except asyncio.TimeoutError:
            if timeout_abort_rid is not None:
                self._abort_request_id(timeout_abort_rid)
                logger.warning(
                    "gRPC chunk backpressure wait timed out after %ss; aborted request",
                    self._BACKPRESSURE_TIMEOUT_S,
                )
            else:
                logger.warning(
                    "gRPC chunk backpressure wait timed out after %ss; closing stream",
                    self._BACKPRESSURE_TIMEOUT_S,
                )
            return False
        ready_event.clear()
        return True
```

## Continuous Batching（concept:continuous-batching）

**定义：** Scheduler 动态将 waiting 队列中的请求合并为 prefill/decode batch，GPU 不空闲等待。

**Explain：** 默认 `event_loop_overlap` 用 `result_queue` 把「上一轮 GPU forward 的结果处理」与「当前轮 forward」重叠：CPU 在 pop 上一轮 `process_batch_result` 的同时，GPU 已在跑新 batch。这是 SGLang 高吞吐的关键之一。

**Code：**

```python
## 来源：python/sglang/srt/managers/scheduler.py L1551-L1595
    def event_loop_overlap(self):
        """A scheduler loop that overlaps the CPU processing and GPU computation."""
        self.result_queue: Deque[
            Tuple[ScheduleBatch, Union[GenerationBatchResult, EmbeddingBatchResult]]
        ] = deque()

        def pop_and_process():
            # Process the results of the last batch
            tmp_batch, tmp_result = self.result_queue.popleft()
            self.process_batch_result(tmp_batch, tmp_result)

        while True:
            if self.gracefully_exit:
                break

            # Receive requests
            recv_reqs = self.request_receiver.recv_requests()
            self.process_input_requests(recv_reqs)
            if self._engine_paused:
                continue

            self._apply_war_barrier()

            # Get the next batch to run
            batch = self.get_next_batch_to_run()
            self.cur_batch = batch
            disable_overlap_for_batch = self.is_disable_overlap_for_batch(batch)

            # If we do not need to overlap the current batch with the last batch,
            # we can process the last batch immediately.
            if disable_overlap_for_batch:
                pop_and_process()
                # Opportunistic flush at the disable_overlap sync boundary:
                # forward_stream is idle (prev forward drained, next not launched),
                # so `_flush`'s non-urgent guard compacts freely. Sync-free, best-effort.
                if self.server_args.enable_unified_memory:
                    try:
                        self.token_to_kv_pool_allocator.flush_opportunistic()
                    except Exception:
                        pass

            # Launch the current batch
            if batch:
                batch_result = self.run_batch(batch)
                self.result_queue.append((batch.copy(), batch_result))
```

**Comment：**

- `get_next_batch_to_run` 内 prefill 优先于 decode；显存不足时 `retract` 把 decode 请求退回 waiting。
- PD / PP / MLX 等模式会走 `dispatch_event_loop` 的其他循环变体。

**专题：** 07–08 | **层：** 请求调度

## Forward Mode（concept:forward-mode）

**定义：** `ForwardBatch` 用枚举区分 prefill、decode、投机、PD、chunked prefill 等前向语义；ModelRunner 与 Attention backend 据此选 metadata 与 kernel。

**Explain：** 常见路径：`EXTEND`（prefill 新 token）、`DECODE`（单步）、`MIXED`（chunked prefill 同批）。高级特性扩展：`TARGET_VERIFY`/`DRAFT_EXTEND_V2`（投机）、`PREBUILT`（PD decode 侧跳过 prefill）。

**Code：**

```python
## 来源：python/sglang/srt/model_executor/forward_batch_info.py L78-L102
class ForwardMode(IntEnum):
    # Extend a sequence. The KV cache of the beginning part of the sequence is already computed (e.g., system prompt).
    # It is also called "prefill" in common terminology.
    EXTEND = auto()
    # Decode one token.
    DECODE = auto()
    # Contains both EXTEND and DECODE when doing chunked prefill.
    MIXED = auto()
    # No sequence to forward. For data parallel attention, some workers will be IDLE if no sequence are allocated.
    IDLE = auto()

    # Used in speculative decoding: verify a batch in the target model.
    TARGET_VERIFY = auto()
    # Used in speculative decoding: extend a batch in the draft model.
    DRAFT_EXTEND_V2 = auto()

    # Used in disaggregated decode worker
    # Represent a batch of requests having their KV cache ready to start decoding
    PREBUILT = auto()

    # Split Prefill for PD multiplexing
    SPLIT_PREFILL = auto()

    # Used in dLLM
    DLLM_EXTEND = auto()
```

**Comment：**

- `ScheduleBatch` 在组 batch 时设置 mode；`ForwardBatch.init_new` 拷贝到 GPU 侧。
- `is_prefill()` / `is_decode()` 封装判断，避免 Scheduler 与 ModelRunner 重复分支逻辑。

## Prefix Cache（concept:prefix-cache）

**定义：** 跨请求共享相同 prompt 前缀的 KV slot，由 RadixCache 树结构索引。

**Explain：** 请求入队时 `match_prefix` 沿 radix 树查找最长命中前缀，返回已缓存 KV 的 pool indices 写入 `req.prefix_indices`；extend 阶段只对未命中 token 做 prefill。请求完成或 chunked prefill 中间态时 `insert` 把新 token→KV 映射挂回树。`extra_key` 隔离 LoRA / 不同 cache 命名空间。

**Code：**

```python
                empty result with the root as the last node.

        Returns:
            MatchResult: ``device_indices`` is a 1-D ``torch.int64`` tensor of
            the concatenated KV cache indices corresponding to the longest
            cached prefix (may be length 0).
```

## RadixAttention（concept:radix-attention）

**定义：** 基于 Radix Tree 的前缀 KV 缓存共享机制，SGLang 相对 vLLM 的核心差异化能力。

**Explain：** 模型层 `RadixAttention` 不自己管 cache 树，而是持有 `layer_id`，在 forward 时从全局 Attention backend 取 KV slot 索引。前缀共享发生在 Scheduler 侧的 `RadixCache.match_prefix`，命中后 extend 只算未缓存 token。

## Speculative Decoding（concept:speculative-decoding）

**定义：** Draft 模型快速提议 token 序列，Target 模型并行验证，接受则加速 decode。

**Explain：** `EAGLEWorkerV2` 包装 target `TpModelWorker` 与独立 draft worker。Prefill 阶段 target 先 extend 并捕获 hidden states，再 draft extend 对齐 KV；decode 阶段交替 draft 多步与 target verify，reject sampling 决定接受长度。

**Code：**


## PD Disaggregation（concept:pd-disaggregation）

## ModelRegistry 机制

**定义：** 启动时扫描 `sglang.srt.models` 包，将 HuggingFace `config.architectures` 映射到 Python 模型类。

**Explain：** `resolve_model_cls` 按 architecture 字符串顺序尝试加载已注册 EntryClass；首个命中即返回。新模型通常只需新增 `srt/models/xxx.py` 并在包 `__init__` 中触发 register。

**Code：**

```python
## 来源：python/sglang/srt/models/registry.py L80-L91
    def resolve_model_cls(
        self,
        architectures: Union[str, List[str]],
    ) -> Tuple[Type[nn.Module], str]:
        architectures = self._normalize_archs(architectures)

        for arch in architectures:
            model_cls = self._try_load_model_cls(arch)
            if model_cls is not None:
                return (model_cls, arch)

        return self._raise_for_unsupported(architectures)
```

## LazyImport Engine 陷阱

**定义：** `from sglang import Engine` 得到 Runtime Engine 而非 Frontend Engine。

**Explain：** 顶层 `__init__.py` 用 `LazyImport` 延迟加载 srt 侧 `Engine`，避免 import sglang 时拉起整个 CUDA runtime。写脚本或 notebook 时若混淆 Frontend `Engine`（lang 包）与 Runtime `Engine`（srt entrypoints），会出现「以为在调编程接口，实际已 spawn 子进程」的错配。

**Code：**

```python
## 来源：python/sglang/__init__.py L79
Engine = LazyImport("sglang.srt.entrypoints.engine", "Engine")
```
