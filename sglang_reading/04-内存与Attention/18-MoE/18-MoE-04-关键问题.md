# MoE：关键问题

## Q1：Router softcapping 作用？

**Explain：** 训练时 gate logits 可能过大导致少数 expert 垄断；softcapping 用 tanh 变体 `top/bottom * cap` 限制幅度，推理沿用相同公式保持分布一致。`moe_softcapping=0` 时跳过，直接使用原始 logits。

**Code：**

```python
# 来源：python/sglang/srt/layers/moe/router.py L47-L55
    # logit softcap
    if moe_softcapping == 0:
        logits_softcapped = logits
    else:
        logits_scaled = logits / moe_softcapping
        exped = tl.exp(2 * logits_scaled)
        top = exped - 1
        bottom = exped + 1
        logits_softcapped = top / bottom * moe_softcapping
```

```python
# ❌ 误以为 softcapping 是 softmax——它是 tanh 变体，在 topk 之前
# ✅ correction_bias 在 softcap 之后叠加，两者独立
```

## Q2：EP 下 dispatch/combine 为何是瓶颈？

**Explain：** Expert Parallel 将 expert 权重切分到多 rank，每个 token 的 topk expert 可能分布在不同 rank；dispatch 需 all-to-all 发送 token 到目标 rank，combine 再 all-to-all 收回结果。通信量与 `bs × topk × hidden_dim` 成正比，远超 router 的轻量 GEMV。DeepEP 针对 NVLink 拓扑优化，Mooncake/NIXL 面向 RDMA 集群。

**Code：**

```python
# 来源：python/sglang/srt/layers/moe/utils.py L539-L548
# Unit of padding - context dependent
def get_moe_padding_size(is_aiter_moe):
    if is_aiter_moe:
        return AITER_PADDING_SIZE
    else:
        return (
            TRITON_PADDING_SIZE
            if bool(int(os.getenv("SGLANG_MOE_PADDING", "0")))
            else 0
        )
```

## Q3：EPLB 何时触发 rebalance？

**Explain：** `EPLBManager._entrypoint` 是 generator，每 `_rebalance_num_iterations` 次 forward yield 后调用 `rebalance()`。rebalance 读取 `expert_distribution_recorder` 的 token 计数，重排 logical→physical 映射并更新权重。buffer size 必须 ≤ rebalance 间隔，否则统计含 stale 数据。

**Code：**

```python
# 来源：python/sglang/srt/eplb/eplb_manager.py L172-L198
 assert (
 self._server_args.eplb_rebalance_num_iterations
 >= self._server_args.expert_distribution_recorder_buffer_size
 )
 def _entrypoint(self):
 while True:
 for _ in range(self._rebalance_num_iterations):
 yield
 yield from self.rebalance()
```

## Q4：FusedMoE 与 quant_method 如何协作？

**Explain：** `FusedMoE.__init__` 从 QuantizationConfig 获取 `quant_method`（Quantization）；`run_moe_core` 调用 quant_method 创建的 MoeRunner 执行 GEMM。无量化时走 `UnquantizedMoEMethod` + Triton fused_moe；FP8 时 dispatch 到 DeepGEMM/FlashInfer 等 backend。

**Code：**

```python
# 来源：python/sglang/srt/layers/moe/fused_moe_triton/layer.py L1134-L1144
    def forward_impl(self, hidden_states: torch.Tensor, topk_output: TopKOutput):
        origin_hidden_states_dim = hidden_states.shape[-1]
        assert self.quant_method is not None

        dispatch_output = self.dispatcher.dispatch(
            hidden_states=hidden_states, topk_output=topk_output
        )

        combine_input = self.run_moe_core(
            dispatch_output=dispatch_output,
        )
```

## Q5：topk=1 vs topk=2 路径差异？

**Explain：** topk=1 时 router kernel 只写 top1 id 与 invsumexp 权重；topk≥2 时额外 mask top1 再 argmax top2，权重为 `exp(top2-top1)*invsumexp`。combine 阶段对两个 expert 输出分别加权求和。DeepSeek-V2/V3 等模型使用 topk=2 + shared expert。

**Code：**

```python

            axis=0,
```

## Q6：Piecewise CUDA Graph 对 MoE 的影响？

**Explain：** CUDA Graph capture 要求算子序列固定；MoE 层注册为 `moe_forward_piecewise_cuda_graph_impl` custom op，将 dispatch+GEMM+combine 作为 graph 节点。topk 格式须为 standard 或 bypassed，否则 fallback 到 eager `forward_impl` 导致 graph break。

**Code：**

```python
# 来源：python/sglang/srt/layers/moe/fused_moe_triton/layer.py L1107-L1130
        if is_in_tc_piecewise_cuda_graph():
            if TopKOutputChecker.format_is_standard(topk_output):
                return moe_forward_piecewise_cuda_graph_impl(
                    hidden_states,
                    topk_output.topk_weights,
                    topk_output.topk_ids,
                    topk_output.router_logits,
                    self.layer_id,
                )
            elif TopKOutputChecker.format_is_bypassed(topk_output):
                return fused_moe_bypassed_piecewise_cuda_graph_impl(
                    hidden_states,
                    topk_output.router_logits,
                    topk_output.topk_config.top_k,
                    topk_output.topk_config.topk_group,
                    topk_output.topk_config.num_expert_group,
                    topk_output.topk_config.correction_bias,
                    topk_output.topk_config.renormalize,
                    self.layer_id,
                    topk_output.topk_config.allow_routed_experts_capture,
                )
            else:
                # Make sure there is torch lib op registration for the whole moe layer
                return self.forward_impl(hidden_states, topk_output)
```

## 验证建议（零基础可试）

1. **启动 MoE 模型并看路由日志** 
 - 操作：`sglang serve --model-path <Mixtral或DeepSeek-MoE> --tp-size 1` 
 - 预期：启动日志出现 MoE / expert 相关初始化；无 `Model not found`。 
 - 对应：本模块 README · [[18-MoE-01-核心概念|01-核心概念 §1]]

2. **对比 EP 开/关（若有多卡）** 
 - 操作：同一 prompt 分别 `--ep-size 1` 与 `--ep-size 2` 压测 decode QPS。 
 - 预期：小 batch 时 EP 未必更快；观察 NVLink/RDMA 利用率。 
 - 对应：[[18-MoE-03-数据流与交互|03-数据流与交互]]

3. **Profiler 定位瓶颈阶段** 
 - 操作：用 nsys 或 PyTorch profiler 抓单次 forward 中 MoE 子模块耗时。 
 - 预期：能区分 Router / dispatch / GEMM / combine 占比。 
 - 对应：用户故事「all-to-all 打满」
