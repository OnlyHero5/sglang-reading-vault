# 启动链路：关键问题

> FAQ、易错点、与旧入口/其他框架的对比

---

## 1. 为什么 `sglang serve` 比 `python -m sglang.launch_server` 更好？

**Explain：** 旧入口缺少 diffusion 自动检测、双层 help 展示，且会打印 deprecation warning。新入口在 `cli/serve.py` 中统一处理模型类型分发与进程清理。

**Code（旧入口 — 不推荐）：**

```python
## 来源：python/sglang/launch_server.py L54-L61
# 提交版本：70df09b
if __name__ == "__main__":
    warnings.warn(
        "'python -m sglang.launch_server' is still supported, but "
        "'sglang serve' is the recommended entrypoint.\n"
        "  Example: sglang serve --model-path <model> [options]",
        UserWarning,
        stacklevel=1,
    )
```

**Code（新入口 — 推荐）：**

```python
## 来源：python/sglang/cli/serve.py L121-L128
# 提交版本：70df09b
        else:
            # Logic for Standard Language Models
            from sglang.launch_server import run_server
            from sglang.srt.server_args import prepare_server_args

            server_args = prepare_server_args(dispatch_argv)

            run_server(server_args)
```

**Comment：**

- 新入口多了 `_extract_model_type_override`、diffusion 检测、双层 help。
- 两者最终都调用 `prepare_server_args` + `run_server`，LLM 路径等价。
- 旧入口仍可用于脚本/CI 中直接传 `sys.argv`，但不建议新代码使用。

## 2. `--model-path` 放在哪？为什么 help 不显示完整参数？

**Explain：** `cli/main.py` 的 parser 不注册 `--model-path`（`add_help=False`），所有 serve 参数通过 `extra_argv` 传递。因此 `sglang serve --help` 需要 `serve.py` 自行展示 help。

**Code（易错：直接在 main parser 上找参数 — 找不到）：**

```python
## 来源：python/sglang/cli/main.py L17-L21
# 提交版本：70df09b
    subparsers.add_parser(
        "serve",
        help="Launch an SGLang server.",
        add_help=False,
    )
```

**Code（正确：serve.py 自行处理 help）：**

```python
## 来源：python/sglang/cli/serve.py L50-L68
# 提交版本：70df09b
    if any(h in extra_argv for h in ("-h", "--help")):
        # Since the server type is determined by the model, and we don't have a model path,
        # we can't show the exact help. Instead, we show a general help message and then
        # the help for both possible server types.
        print(
            "Usage: sglang serve --model-path <model-name-or-path> [additional-arguments]\n\n"
            "This command can launch either a standard language model server or a diffusion model server.\n"
            "The server type is determined by the --model-path.\n"
            "Optional override: --model-type {auto,llm,diffusion} "
            "(default: auto, fallback to LLM on detection failure)."
        )

        print("\n--- Help for Standard Language Model Server ---")
        from sglang.srt.server_args import prepare_server_args

        try:
            prepare_server_args(["--help"])
        except SystemExit:
            pass  # argparse --help calls sys.exit
```

**Comment：**

- 英文 help 的核心意思：`sglang serve` 根据 `--model-path` 判断启动 LLM 服务还是 diffusion 服务；如需绕过自动判断，可显式传 `--model-type`。
- 用户常犯 Mistake：运行 `sglang --help` 看不到 `--model-path`，需要 `sglang serve --help`。
- help 会分别展示 LLM（ServerArgs 全量参数）和 diffusion 两套参数列表。
- 不带 `--model-path` 启动会直接报错（`get_model_path` 抛 Exception）。

## 3. `ServerArgs` 有数百个字段，如何快速定位？

**Explain：** 字段按 comment banner 分组（Model、HTTP server、LoRA、Disaggregation 等）。CLI flag 从字段名自动推导，可用 `sglang serve --help | grep keyword` 搜索。

**Code（字段分组示例）：**

```python
## 来源：python/sglang/srt/server_args.py L416-L418, L527-L529, L2378-L2384
# 提交版本：70df09b
    # -------------------------------------------------------------------------
    # Model and tokenizer
    # -------------------------------------------------------------------------
```

**Comment：**

- 新增参数的标准流程：在对应 section 加 Annotated 字段 → 自动出现在 CLI。
- 仅 deprecated flags、动态 choices、`--config` 需要手动改 `add_cli_args`。
- 完整字段列表太长，本模块文档只覆盖启动相关字段；调度/内存/Attention 字段在后续专题按需引用。

## 4. 插件加载失败会影响启动吗？

**Explain：** 单个插件 load/execute 失败会被 catch 并 log，**不会阻断**其他插件或主启动流程。但 Hook apply 失败可能导致部分 monkey-patch 未生效。

**Code：**

```python
## 来源：python/sglang/srt/plugins/__init__.py L79-L84, L133-L136
# 提交版本：70df09b
        try:
            func = ep.load()
            plugins[ep.name] = (func, dist_name)
            logger.info("Loaded plugin %s from group %s", ep.name, group)
        except Exception:
            logger.exception("Failed to load plugin %s from group %s", ep.name, group)
```

**Comment：**

- 排查插件问题：设置 `SGLANG_LOG_LEVEL=DEBUG` 或查看启动日志中的 `Loaded plugin` / `Failed to load` 行。
- 用 `SGLANG_PLUGINS=plugin_a,plugin_b` 限制加载范围，避免无关插件干扰。
- 用 `SGLANG_PLATFORM=xpu` 等平台变量自动排除其他硬件平台的插件包。

## 5. `--config` YAML 与 CLI 参数的优先级？

**Explain：** 当 argv 含 `--config` 时，`ConfigArgumentMerger` 先读 YAML 再与 CLI 合并——**CLI 显式传入的参数覆盖 config 文件**。

**Code：**

```python
## 来源：python/sglang/srt/server_args.py L7575-L7582
# 提交版本：70df09b
    # Check for config file and merge arguments if present
    if "--config" in argv:
        # Import here to avoid circular imports
        from sglang.srt.server_args_config_parser import ConfigArgumentMerger

        # Extract boolean actions from the parser to handle them correctly
        config_merger = ConfigArgumentMerger(parser)
        argv = config_merger.merge_config_with_args(argv)
```

**Comment：**

- 典型用法：YAML 存生产默认配置，CLI 只传 `--model-path` 和少量 override。
- `--config` 本身不是 `ServerArgs` 字段，不会出现在 dataclass 中。
- config 文件必须是 YAML 格式；具体 schema 与字段名对应 CLI flag（去掉 `--` 前缀，`-` 换 `_`）。

## 6. 与 vLLM `--serve` 启动方式的对比

| 维度 | SGLang | vLLM（参考） |
|------|--------|-------------|
| 入口命令 | `sglang serve` | `vllm serve` / `python -m vllm.entrypoints.openai.api_server` |
| 配置载体 | `ServerArgs` dataclass（单文件 7700+ 行） | `AsyncEngineArgs` + argparse |
| 启动分发 | HTTP / gRPC / Ray / Encoder 四路 | 主要是 HTTP OpenAI API |
| 扩展机制 | setuptools entry_points + HookRegistry | 较少原生插件支持 |
| 模型类型 | LLM + diffusion 自动检测 | 仅 LLM |

**Explain：** SGLang 的 `ServerArgs` 设计更「重」——把模型、调度、内存、Attention、Disaggregation 等全部参数收敛到一个 dataclass，并通过 `__post_init__` 做交叉校验。这使得单文件很大，但 CLI 与内部 API 共享同一配置对象，减少不一致。

**Code（SGLang 配置入口）：**

```python
## 来源：python/sglang/srt/server_args.py L7561-L7595（核心 3 行）
# 提交版本：70df09b
def prepare_server_args(argv: List[str]) -> ServerArgs:
    """
    Prepare the server arguments from the command line arguments.

    Args:
        args: The command line arguments. Typically, it should be `sys.argv[1:]`
            to ensure compatibility with `parse_args` when no arguments are passed.

    Returns:
        The server arguments.
    """
    parser = argparse.ArgumentParser(prog="sglang serve")
    ServerArgs.add_cli_args(parser)

    # Check for config file and merge arguments if present
    if "--config" in argv:
        # Import here to avoid circular imports
        from sglang.srt.server_args_config_parser import ConfigArgumentMerger

        # Extract boolean actions from the parser to handle them correctly
        config_merger = ConfigArgumentMerger(parser)
        argv = config_merger.merge_config_with_args(argv)

    raw_args = parser.parse_args(argv)

    # Set up basic logging before ServerArgs.__post_init__ so that
    # logger.info / logger.warning calls there are properly formatted.
    logging.basicConfig(
        level=getattr(logging, raw_args.log_level.upper()),
        format="[%(asctime)s] %(message)s",
        datefmt="%Y-%m-%d %H:%M:%S",
        force=True,
    )

    return ServerArgs.from_cli_args(raw_args)
```

**Comment：**

- vLLM 用户迁移时，大部分 flag 名称相似（`--tensor-parallel-size` ↔ `--tp-size`），但 SGLang 有更多特有参数（RadixAttention、PD 分离、投机解码等）。
- 不要假设 vLLM 的默认值与 SGLang 相同——务必查阅 `sglang serve --help`。

## 7. 常见问题速查

| 问题 | 原因 | 解决 |
|------|------|------|
| `Error: --model-path is required` | 未传模型路径 | 加 `--model-path <path>` |
| `sglang serve --help` 无输出 | 用了 `sglang --help` | 改用 `sglang serve --help` |
| 端口被占用 | 默认 30000 | `--port <other>` |
| Ray 模式 ImportError | 未装 ray extra | `pip install 'sglang[ray]'` |
| diffusion 误判为 LLM | 检测 fail-open | 加 `--model-type diffusion` 强制 |
| 插件 Hook 未生效 | apply 在 import 之后 | 确保 `load_plugins()` 在 Runtime import 前 |

## 验证建议（零基础可试）

以下 **4 条不要求 GPU**，可在读完本模块 FAQ 后立刻动手，确认 CLI 入口与 `ServerArgs` 行为与文档一致。

1. **操作：** 分别执行 `sglang --help` 与 `sglang serve --help`，对比输出。 
 **预期现象：** 前者只有 `serve` / `generate` 等子命令，**没有** `--model-path`；后者开头为 `Usage: sglang serve --model-path <model-name-or-path> ...`，并展开 LLM / diffusion 两套参数列表。 
 **对应文档节：** §2 `--model-path` 放在哪、§7「`sglang serve --help` 无输出」

2. **操作：** 不带模型路径运行 `sglang serve 2>&1 | head -5`（Windows 可去掉管道，直接看前几行 stderr）。 
 **预期现象：** 报错 `--model-path is required` 或 `get_model_path` 相关 Exception，进程非 0 退出。 
 **对应文档节：** §7 FAQ 表第一行

3. **操作：** `python -W default::UserWarning -m sglang.launch_server --help 2>&1 | head -20` 
 **预期现象：** stderr 出现 deprecation warning，推荐使用 `sglang serve`；help 仍可用。 
 **对应文档节：** §1 新旧入口对比

4. **操作：** `sglang serve --help 2>&1 | rg -i "port|host|tp-size|config"`（无 `rg` 时用 `findstr /i "port host config"`）。 
 **预期现象：** 命中 `--port`（默认 30000）、`--host`、`--tp-size`、`--config` 等 flag，印证 ServerArgs 字段分组与 CLI 自动映射。 
 **对应文档节：** §3 ServerArgs 快速定位、§5 `--config` YAML 优先级
