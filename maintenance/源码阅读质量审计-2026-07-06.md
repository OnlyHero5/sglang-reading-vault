# 源码阅读质量审计

> 历史快照日期：2026-07-06；按当前质量模型复核：2026-07-13。文件名保留快照日期，不把旧时点结论冒充当前发布结论。

## 质量模型

一篇合格的源码讲解应让读者完成五件事：理解为什么要读、建立心理模型、沿真实对象追源码、验证关键判断、把结论迁移到排障或改代码。

## 自动检查

| 检查 | 工具 | 当前目标 |
|------|------|----------|
| 双链与唯一文件名 | `audit_wikilinks.mjs` | 断链与重复名为 0 |
| 源码文件与行号 | `audit_source_evidence.mjs` | 缺失文件与越界范围为 0 |
| Properties、H1、Mermaid、阅读信号 | `audit_markdown_quality.mjs` | errors 为 0，warnings 持续下降 |
| 空白与冲突标记 | `git diff --check` | 无格式错误 |

## 人工顺读

自动检查不能判断讲解是否真的好读。日常维护至少抽样；宣称全库达到出版级质量时，当前可写范围必须逐篇全文终审，不能以抽样替代：

- 首页到课程主线是否能在两分钟内选路。
- 专题入口是否先回答读者任务，再给文档分工。
- 源码走读是否按请求、样本、权重或 tensor 的因果顺序展开。
- 排障是否从症状进入日志、配置、源码入口和预期现象。
- 学习检查是否验收能力，而不是文档数量和阅读篇数。

## 当前风险

- 少数全链路和源码走读超过单次阅读负担，需要依赖“长文读法”和短课程分流。
- 一部分旧深度文档只有验证提示，没有完整的操作与预期对照。
- 历史经验数字需要逐步补来源、环境和证据等级。

## 审计结论

语义命名、统一 Properties、课程层、Bases 和三套自动审计已经建立。后续质量提升应以真实顺读、实验可执行性和证据强度为主，不能退回按固定模板计数的维护方式。

当前逐篇状态与最终发布结论以 [[maintenance/出版级精修台账]] 为准。
