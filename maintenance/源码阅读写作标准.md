# 源码阅读写作标准

## 核心目标

源码讲解的目标不是逐行翻译，而是让读者建立可运行、可验证、可迁移的心理模型。每篇主文档先回答：为什么要读、读完能解决什么问题、贯穿对象是什么、源码如何证明、怎样验证。

整篇采用：**场景主线 → 心理模型 → 源码证据 → 运行验证 → 复盘迁移**。`Explain → Code → Comment` 只可作为局部证据卡的写法，不作为全文目录。

## 读者分层

| 读者任务 | 文档应提供 |
|----------|------------|
| 第一次学习 | 最小术语、主线图、对象生命周期、下一站 |
| 生产排障 | 症状、日志、配置、源码入口、预期现象 |
| 准备改代码 | 不变量、边界、测试、替代设计代价 |

同一篇无法清楚服务三种任务时，拆成专题入口、核心概念、源码走读、数据流、排障或学习检查。不要为了凑齐固定套件而创建空洞文档。

## 推荐结构

```markdown
# 主题 · 源码走读

## 读者任务

## 先建立模型

## 贯穿场景

## 主线走读

## 运行验证

## 失败模式与排障入口

## 复盘
```

标题可以按源码语义继续细分，但不要用内部发布顺序或迁移编号组织读者认知。

## 源码证据卡

每张证据卡只证明一个判断，至少覆盖以下信息中的三类：

- 系统压力：延迟、吞吐、显存、一致性、并发、兼容性或可观测性。
- 设计选择：源码采用了什么结构，代价和替代方案是什么。
- 执行逻辑：关键分支、状态读写、对象形态和调用顺序。
- 不变量与失败模式：破坏约束后会发生什么。
- 读者抓手：如何接回主线，下一步观察哪里。

来源写在代码块首行：

```python
# 来源：python/sglang/example.py L10-L24
```

代码必须逐字来自当前 upstream 基线。不能编造行号、签名、行为或性能结论；设计动机必须能回指初始化顺序、分支条件、数据结构、测试或 upstream 文档。

## 心理模型与图

一张图只回答一个问题，例如请求状态机、样本生命周期、权重同步屏障、Q/K/V tile 的内存移动。类比必须映射到真实源码对象，并说明失效边界。

## 运行验证

核心源码走读、排障和实验至少给出一种“操作 + 预期”：

| 验证方式 | 示例 |
|----------|------|
| 配置对照 | 打开或关闭 overlap、varlen、offload |
| 日志与 metric | TTFT、KV usage、rollout time、retract |
| 最小命令 | `curl`、`pytest`、benchmark、`rg` |
| 断点 | 函数名、触发条件、关键变量 |

只给命令不够；必须说明看到什么表示模型成立，看到什么需要转入排障。

## 长文标准

长文开头必须有“长文读法”，告诉首次读者必读段、可跳过的参考段和回看入口。超过单次阅读负担时，优先把连续学习主线留在短文，把完整证据放入 `learning_role: reference` 文档。

## 学习检查

学习检查以能力为标准：能画出对象和边界、复述生命周期、定位失败模式、完成验证实验、说明配置变化后的分叉。不要检查读过多少篇、贴了多少段源码或记住多少维护顺序。

## 写作前置与审计

修改源码解释前，先完整阅读本文实际引用的 upstream 文件。新增或调整摘录后运行源码证据审计；该审计既检查文件/行号，也逐行比对代码围栏首行标为“来源”的正式源码卡。删节、去注释或拼接后的内容必须标为定位骨架，不能冒充连续原文。移动或重命名后运行双链审计；发布前运行 Markdown 质量审计与 `git -c core.autocrlf=false diff --check`。自动审计只能证明结构和引用下限，不能替代修改后的全文语义复读。

## 方法依据

- [Diátaxis](https://diataxis.fr/)：区分学习、操作、解释和参考任务。
- [Obsidian Help](https://help.obsidian.md/)：使用 Properties、Bases、Backlinks、Graph、Search、Bookmarks 和 Templates 组织知识。
- [Write the Docs: Docs as Code](https://www.writethedocs.org/guide/docs-as-code.html)：版本控制、审查和自动检查。
- [Cognitive Apprenticeship](https://www.aft.org/ae/winter1991/collins_brown_holum)：显式展示建模、脚手架、表达和复盘过程。
- [Program comprehension research](https://www.cs.kent.edu/~jmaletic/cs69995-PC/papers/von_mayrhauser-1995.pdf)：在自顶向下假设、控制流追踪和自底向上证据之间迭代。
